## ‚öÝÔ∏è CRITICAL BUG DISCOVERED

**Jules session page ignores URL `repo=` parameter.** The dropdown reverts to whatever repo was last selected in that browser session. Despite navigating to `?repo=github%2FNixeloApp%2Fnixelo`, the dropdown kept showing StartHub.

### Root Cause
The session page caches the last-used repo in the browser and doesn't update when URL changes.

### Workaround
- **ALWAYS verify the dropdown shows the correct repo before EVERY submission**
- Must manually click dropdown and select correct repo each time
- URL parameter is NOT reliable
- **Better approach**: Use "New" button from repo Overview page (e.g., `/repo/github/NixeloApp/nixelo/overview`) instead of session URL

## API Reverse Engineering (Attempted)

### Findings
- Jules UI uses Google's internal `batchexecute` RPC (NOT a public API)
- Endpoint: `/u/0/_/Swebot/data/batchexecute`
- Uses obfuscated method IDs: `p1Takd` (task listing), `cFjlx`, `c9RQT` (repo info)
- Request format: `f.req` parameter with nested JSON arrays, requires `at` CSRF token
- Session params: `f.sid=-3100894435837849720`

### Conclusion
- No public Jules API for scheduled/recurring tasks
- Jules CLI (`/home/mikhail/.npm-global/bin/jules`) only supports one-off sessions
- **Browser automation is the ONLY way** to create native daily recurring tasks
